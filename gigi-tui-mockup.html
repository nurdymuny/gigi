GQL · template for HUNT
HUNT <pattern> IN <bundle> [ EXCLUDING IN <bundle> ] [ WITH NEAR_MISS_BUDGET k ] [ RANK BY _score DESC ] [ PROJECT (field,…) ] [ TOP n ] ;
DESCRIBES Run a named pattern against a bundle and return ranked rows with a sat/unsat/near-miss verdict.
EXAMPLE
HUNT high_value_churn IN customers EXCLUDING IN converted_customers WITH NEAR_MISS_BUDGET 1 PROJECT (customer_id, _score) TOP 20 ;

5Teach mode · the current GQL, annotated, with relations and alternatives
Press t from any state. The current GQL fills the top of the window with one annotation per line. Beneath it: a plain-English summary, a clause-by-clause breakdown, equivalent ways to write the same thing, and related verbs to try next. The discipline is "the operator never has to leave the tool to learn the language" — every TUI interaction is a chance to read GQL alongside its meaning. Works identically whether the operator is in HUNT, BETTI, DREAM, or BEGIN/COMMIT.

YOUR QUERY · annotated
HUNT near_miss_conversion IN customers verb · run a named pattern against a target bundle, returning ranked rows EXCLUDING IN converted_customers anti-join clause · drop any row whose PK appears in converted_customers WITH NEAR_MISS_BUDGET 1 v0.2 flag · keep rows that would match if 1 field were flipped — verdict becomes near_miss RANK BY _score DESC sort by the WEIGHT-computed _score, highest first (this is the default for HUNT; making it explicit is fine) PROJECT (customer_id, last_seen, cart_value, _score) keep only these columns in the output — saves bandwidth and centers the operator's attention TOP 20 ; limit to the 20 highest-scoring rows (statement terminator is the semicolon)
WHAT THIS QUERY DOES
Look at every row in customers that's one field flip away from satisfying the near_miss_conversion pattern. Drop any of those whose PK already appears in converted_customers. Score the survivors with the pattern's WEIGHT, sort highest-first, and return the top 20 with just customer_id, last_seen, cart_value, and the computed _score.
EQUIVALENT FORMS
  • COVER with a custom WHERE and a hand-rolled sort — you lose the verdict envelope and the WEIGHT-derived _score, but it works for one-off exploration.
  • HUNT without WITH NEAR_MISS_BUDGET — v0.1-compatible mode: returns only strict matches (verdict = sat).
  • HUNT … WITH NEAR_MISS_BUDGET 1 EXPLAIN REPAIR_MENU — adds _explain per row and a flip menu per near-miss.
RELATED VERBS
  • COVER · same target without scoring or patterns — the everyday "show me rows" verb
  • DEFINE PATTERN · edit the pattern this hunt resolves to — change predicate, weight, or relaxation_costs
  • EXISTS · boolean membership test, useful inside WHERE NOT EXISTS (…) sub-clauses
  • SUDOKU · if the pattern is constraint-shaped, SUDOKU exposes the same verdict / repair_menu on free-form constraint sets
