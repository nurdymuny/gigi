The readout is landed. There are two grammar forms, both on the existing `HOLONOMY` keyword (disambiguated from `ON` / `NEAR` by the `AROUND` keyword), and both return exactly one row `{ q0, q1, q2, q3, re_trace, order_estimate, group_used }`. The named-cycle form is `HOLONOMY <field> AROUND CYCLE AXIS <ax> AT (<c0>, <c1>)` — e.g. `HOLONOMY lens_field AROUND CYCLE AXIS z AT (0, 0);` — which enumerates all links along `<ax>` at the two fixed transverse coordinates in +axis order, closing via the wrap edge; it needs the field's bound CUBIC lattice to walk the loop and errors clearly if that binding is missing or non-cubic. The explicit form is `HOLONOMY <field> AROUND CYCLE EDGES (<e0>, <e1>, ...)` — e.g. `HOLONOMY lens_field AROUND CYCLE EDGES (261, 286, 311, 336, 361);` — an ordered edge-id list with no lattice binding required, product taken in list order. `q0..q3` are the SU(2) quaternion of the ordered link product; `re_trace = ½·Tr(U) = q0 = cos` of the half-angle (1 iff identity); on a clean lens space `re_trace = cos(2πq/p)`. `group_used` is `"SU(2)"`.

The direction convention is load-bearing and I have pinned it, so your `pc_gigi_lens_space.py` constructor matches on the first try. Stated verbatim: **AXIS walks +axis order; each edge whose stored lattice direction matches the walk contributes U (Forward), else U† (Reverse); on a periodic cubic the z-wrap link is stored (s, s+ê_z) so the +z walk returns Ω, not Ω†.** Concretely, the AXIS form builds the vertex cycle `[site(t=0), …, site(t=L-1)]` and resolves each consecutive pair (incl. the wrap `site(L-1)→site(0)`) through `resolve_edge`, which returns Forward when the stored edge already runs +axis. So if you store `Ω = exp(2πi·q·σ₃/p)` on the +z link at `(x0,y0)` — stored direction `(s, s+ê_z)` — the loop returns Ω. If your constructor instead stamps Ω on the −z link, you will read Ω† and every class `p` comes back as its inverse class (silently — `re_trace` is even in q, so the *order* is unaffected, only the quaternion axis sign flips). Test H3 is the receipt: reversing a loop (reverse order + inverted links) yields the exact conjugate `(q0, −q1, −q2, −q3)`, and the ordered product is direction-sensitive (`A·B ≠ B·A`). If your first live read of a known-`(p,q)` twist gives you `+q3`, you are aligned; a `−q3` means flip the stored wrap direction.

`order_estimate` is the best-effort nearest integer `p` such that the element has order `p`: for `g = (cos φ, sin φ·n̂)`, `gⁿ = identity` iff `nφ ≡ 0 (mod 2π)`, so `p` is the denominator of `x = arccos(q0)/(2π)` in lowest terms. I recover it by continued-fraction (Stern–Brocot) rational approximation of `x` with absolute tolerance `TOL = 1e-9` and denominator cap `Q_MAX = 512`; identity (`q0 ≥ 1 − TOL`) returns 1. It is branch-robust — `arccos` folds `φ ↔ 2π−φ` but the denominators of `x` and `1−x` are identical, so `order = p/gcd(p,q) = p` when `gcd(q,p) = 1`. Honest caveat: it is best-effort, tuned to nail `p ∈ {2,3,5,7}` (and `q ∈ {1,2,3}`) cleanly on the acceptance fixtures, not to be a robust order-finder for arbitrary noisy holonomies; you can always re-derive it client-side from `re_trace`, so treat the field as a convenience. One more caveat worth stating plainly: `order_estimate` is only *meaningful* on a clean lens wrap (a pure σ-twist). Fed a generic SU(2) holonomy it still returns a bounded integer (`≤ Q_MAX = 512`) rather than a distinguished sentinel — so there is no in-row flag separating "clean order-`p` element" from "generic element, order is noise" beyond `re_trace` itself. Read it only alongside a `re_trace` you already expect to be `cos(2πq/p)`; on your lens fixtures that pairing is exactly what you have.

The receipt, at `p = 5`: the synthetic lens field your `pc_gigi_lens_space.py` will emit — `Ω = (cos 2π/5, 0, 0, sin 2π/5)` on the z-wrap of an `L=5, DIM=3` periodic cubic, identity elsewhere — reads `re_trace = cos(2π/5) = 0.309017`, `q3 = +sin(2π/5) = 0.951057`, `order_estimate = 5`, `group_used = "SU(2)"`. That number is produced by acceptance tests H2 (`(p,q)=(5,1)`, also `(2,1),(3,1),(5,2),(7,1),(7,3)` — order tracks `p` in every case, `re_trace` to 1e-12) and H4 (the AXIS form at `(x0,y0)` equals the closed-form explicit EDGES list `eid = 2·V + site(x0,y0,z)`, so the lattice enumeration is verified against the cubic edge-layout formula). The end-to-end live probe against gigi-stream (build the twisted-BC field, read `AXIS z`, confirm `0.309017 / 5`) is the ship agent's step; the verb behind it is what this note describes.
